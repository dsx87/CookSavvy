# CookSavvy — Senior Engineering Audit

**Date:** 2026-06-06
**Scope:** Full codebase (222 Swift files, ~26k LOC) across 11 dimensions
**Method:** 11 parallel domain experts + adversarial verification of every blocker/high candidate (29 agents total). Severities below are the **post-verification** values — several agent-claimed severities were corrected up or down after a second agent re-read the actual code. Headline blockers were additionally hand-verified against source.

> **How to read severity:** `blocker` = a real defect/policy violation that should gate release (crash, data corruption, monetization hole, security/privacy violation, store rejection). `high` = fix before or immediately after launch. `medium`/`low` = scheduled tech debt. Medium/low findings were **not** independently re-verified (sampling only) — treat them as leads, not confirmed.

---

## Executive Summary

The app is in good architectural shape — MVVM+Coordinator is followed consistently, services sit behind protocols, DI is centralized, and the project builds clean with no concurrency warnings. The codebase is well-tested in breadth (~45 test files) but has **dangerous gaps in depth** on exactly the systems where bugs are most expensive: subscriptions, startup data import, and the database cache.

**The big picture:**
- **1 confirmed startup-crash blocker** (force-unwrap in `ImageService`).
- **2 monetization/security issues** that I'd gate release on for a paid app (auth-error logging → Sentry; client-only scan quota).
- A cluster of **database cache-invalidation and silent-decode-failure** bugs that risk serving stale/incomplete data.
- A cluster of **StoreKit trial-state** bugs that mis-report trial status and can hide the "7 days free" badge.
- **Zero direct unit tests** on the three most critical services: `StoreKitSubscriptionService`, `DataImportService`, `DatabaseInitializationService`.

## Ship Blockers

These should be fixed (or consciously accepted by the business with sign-off) before shipping.

### SB-1 — App-launch crash: force-unwrap of documents directory `[crashrisk]` — ✅ FIXED 2026-06-07
> **Status:** `ImageService.init` is now `throws` with a `guard let … else { throw .documentsDirectoryUnavailable }`; `AppContainer` calls `try ImageService()`, surfacing failures via the existing blocking startup screen. See `docs/APPSTORE_BLOCKERS_STATUS_2026-06-07.md`.

**`CookSavvy/Services/Image/ImageService.swift:123`** *(hand-verified)*
```swift
self.imagesDirectory = fileManager.urls(for: .documentDirectory, in: .userDomainMask).first!
```
`ImageService` is constructed during `AppContainer` init at startup. `init()` is **non-throwing**, so a nil `.first` is an uncatchable crash that bypasses the app's "blocking startup error" handling (which only catches *thrown* errors). The API has no non-empty contract. Other call sites in the same service already use `guard let` (e.g. `ImageExtractor:50`), so this is an inconsistency, not a constraint.
**Fix:** make the path resolution `guard let … else { throw }` and let `AppContainer`'s existing throwing init surface it as the blocking startup screen.

### SB-2 — Auth errors logged verbatim and forwarded to Sentry `[security]` — ✅ FIXED 2026-06-07
> **Status:** All 6 raw-error log sites in `SupabaseAuthService` (lines 113/128/158/170) and `SignInWithAppleAction:125` now log `type(of: error)` instead of the raw error, so no auth payload reaches the crash sink. See `docs/APPSTORE_BLOCKERS_STATUS_2026-06-07.md`.

**`CookSavvy/Services/Auth/SupabaseAuthService.swift:158`** *(hand-verified — note: the security agent's original line cite was wrong; this is the real sink)*
```swift
} catch {
    logger.error("Apple sign-in failed: \(error)")   // implicit String(describing:)
    throw AuthError.signInFailed
}
```
`logger.error` routes into `LoggingService`, whose `crashSink` records a `LoggedFailure` to **Sentry** in RELEASE when a DSN is set. The thrown value here is the **raw Supabase SDK error**, which can embed JWTs, nonces, identity-token fragments, or OIDC endpoint responses. This is both a privacy risk and a direct conflict with the org policy prohibiting logging of sensitive auth data to non-authorized destinations.
**Fix:** log `type(of: error)` (or a curated, field-allowlisted message) — never the raw error description — at this and any other auth/network catch site that feeds the logger. Audit other `logger.error("...: \(error)")` sites in `Services/Auth` and `Services/Supabase`.

### SB-3 — Free-tier camera scan quota is client-only and trivially bypassable `[storekit]` — ⛔ OPEN (backend / business decision)
> **Status:** Not addressed — requires server-side enforcement in the `detect-ingredients` edge function (out of this iOS repo). Remains a conscious launch trade-off pending a business decision.

**`CookSavvy/Services/Subscription/CameraScanTracker.swift`** (all) + gating in `DiscoverViewModel.showCamera()`
The 5-scans/week free limit lives entirely in `UserDefaults` (`camera_scans_used_this_week`, `camera_scan_week_start`). Deleting app data, or editing the key, resets it. The `detect-ingredients` Supabase edge function performs **no server-side quota enforcement** — it only checks subscription auth. A free user gets unlimited AI vision scans for free, defeating a core paywall lever.
This is already documented as Phase-1 backend debt (`docs/BACKEND_PLAN.md:291`), so it may be a conscious launch trade-off — but it **is** a monetization hole and should be an explicit business decision, not an oversight.
**Fix:** enforce the weekly quota server-side in the edge function (rate-limit table keyed by `user_id + week`); treat the client `UserDefaults` count as a cache only.

## High Priority

### Database integrity

**H-1 — Recipe cache not invalidated on delete `[database]` — `DBInterface.swift:1454` (`removeRecipes(withTitles:)`)** *(hand-verified)*
The private `removeRecipes(withTitles:)` (called by `removeRecipes(_:)` and `removeRecipe(withTitle:)`), plus `removeAllRecipes()` and `removeRecipes(byIngredients:)`, delete rows without calling `removeCachedRecipe(forTitle:)`. `updateUserRecipe` (line 1089) does it correctly. Delete-then-reinsert-same-title before FIFO eviction returns the **stale** cached recipe. (`deleteUserRecipe` at 1094 has the same gap but is lower-impact — the cache is title-keyed and the public path doesn't expose a title getter.)
**Fix:** invalidate cache entries for every removed title; centralize so all delete paths share one invalidation step.

**H-2 — Silent JSON decode failures drop recipes without a trace `[database]` — `DBInterface.swift:755, 787, 1026`**
`getRecentRecipes`, `getFavoriteRecipes`, `getUserCreatedRecipes` use `compactMap { try? decodeRecipe(...) }`. A recipe with malformed `instructions_json`/`ingredients_json`/`additional_info_json` silently vanishes from results — no log, no telemetry — masking data corruption. Contrast `getRecipe(byID:)`/`getAllRecipes()` which propagate. Same pattern in `getRecentSearches`/`getCookingSessions` (837, 922).
**Fix:** log decode failures with recipe id/title + error before discarding; consider a telemetry counter so corruption is detectable in production.

### StoreKit / monetization

**H-3 — Monthly trial eligibility can be cached as "ineligible" if products aren't loaded `[storekit]` — `StoreKitSubscriptionService.swift:229, 316–322`**
`loadProducts()` swallows failures (empty products dict). `updateSubscriptionStatus()` then calls `loadMonthlyTrialEligibility()` regardless; with the product missing it `guard`s to `false` (no log/retry), and that `false` is **published to the UI and cached to UserDefaults**. `restorePurchases()` also calls `updateSubscriptionStatus()` without ensuring products are loaded, so it can't self-heal. Net effect: a network hiccup at first launch hides the "7 days free" badge until app restart — a direct conversion hit.
**Fix:** ensure products are loaded before computing eligibility; if the product is nil, publish an "unknown" state and defer to a background refresh rather than caching `false`.

**H-4 — Stale `isOnFreeTrial` after trial→paid conversion `[storekit]` — `StoreKitSubscriptionService.swift:325–339, 353–359`** *(verifier corrected to medium; grouped here as it shares a fix with H-3)*
`isMonthlyFreeTrial()` decides trial state purely from the transaction's offer type and **never checks `expirationDate > now`**. `preferredStatus()` prefers a candidate with `isOnFreeTrial == true`, so an expired trial transaction lingering in `currentEntitlements` keeps the user shown as "on trial." Affects messaging/analytics, not feature gating (gating keys off `plan`).
**Fix:** add `transaction.expirationDate > Date()` to the trial check; deprioritize expired-trial candidates in `preferredStatus()`.

### Test coverage on critical paths

These are **risk**, not runtime defects — but on these three services the absence of tests is itself a launch hazard, because the failure modes are silent (empty DB, wrong entitlement) and unrecoverable in the field. The verification agents argued for upgrading two of these to blocker; I'm keeping them High and flagging the disagreement.

**H-5 — `DataImportService` has zero tests `[tests]`** — mission-critical startup path: unzip dataset, bulk-insert ~1k+ recipes, normalize ingredients, dedup `basicComponent`. A failure silently leaves a first-launch user with an empty/partial recipe catalogue. `DatabaseInitializationServiceTests` uses a no-op mock, so the real path is never exercised. *(verifier wanted blocker)*

**H-6 — `StoreKitSubscriptionService` has zero direct tests `[tests]`** — 386 lines of app-critical entitlement logic (transaction verification `checkVerified`, listener lifecycle, refresh coalescing, cache load/persist, trial eligibility) tested only via the mock. H-3/H-4 are exactly the kind of bug this gap lets through.

**H-7 — `DatabaseInitializationService` effectively untested `[tests]`** — the only test asserts a DEBUG-only `markReadyForTesting()`. The startup state machine, `waitForRecipes()`/`waitForIngredients()` gating, and `.failed` error transitions are untested. *(verifier wanted blocker)*

## Recommended Order of Attack

1. **SB-1** (1-line fix, removes a launch crash) — do today.
2. **SB-2** (auth logging → Sentry) — small fix, org-policy + privacy.
3. **H-1 / H-2** (DB cache invalidation + silent decode logging) — data integrity.
4. **H-3 / H-4** (trial eligibility/state) — protect conversion; pairs naturally with **H-6** (add the StoreKit tests while you're in there).
5. **SB-3** (server-side scan quota) — needs backend work; schedule with the BACKEND_PLAN Phase-1, get a business decision on launch-without-it.
6. **H-5 / H-7** (DataImport + DatabaseInit tests) — close the silent-startup-failure blind spot.
7. Medium/low as scheduled tech debt; **M-1** (DiscoverViewModel) is the largest structural item and worth a dedicated refactor ticket.

## What's Healthy (no action)

- MVVM+Coordinator separation is consistently applied; views are thin.
- Services are protocol-backed and mockable; DI via `AppContainer` is clean; throwing startup with a blocking error screen is the right call.
- Builds with **zero concurrency warnings**; `@MainActor` usage on ViewModels is correct.
- DB access is thread-safe via GRDB `DatabasePool`; migrations are ordered.
- Large dataset correctly handled via Git LFS.
- Good test breadth across ViewModels, ranking, smart search, and ingredient/recipe services — the gaps are specific, not systemic.
